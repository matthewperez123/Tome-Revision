### Demo-readiness rationale

What works, on a happy-path live demo:
- Sign up, verify email, complete onboarding, land on `/dashboard`
- Browse `/library`, open one of the four Tier 1 books (Frankenstein, Meditations, Pride and Prejudice, The Odyssey) and read full text
- Click an annotation marker → annotation drawer opens with curated commentary + cross-references → bookmark the annotation
- Run a Trial (3-tier difficulty popover, 7 question types incl. reflection) and see the polished results screen
- Visit `/bookmarks`, `/friends` (with mock data), `/seals`, marketing pages `/`, `/readers`, `/educators`
- View the Stoa painting gallery (67 paintings, 1:1 invariant enforced)

What does NOT work or is visibly mocked:
- "Talk to Virgil" is a disabled chat footer (Virgil chat is an explicit stub — F4.1)
- Reflection grading returns canned feedback (placeholder per F4.2)
- Trial attempts auto-save to localStorage only (no cross-device, no teacher visibility — F4.3)
- The classroom detail page (`/classroom/[id]`) renders **mock demo data**, not the real classroom — the spec'd Stream/Assignments/Roster/Gradebook tabs don't exist (F5.1, F5.2)
- The notification bell wired to the topbar doesn't render accept/reject buttons for book recommendations (F5.4)
- Bookmarks and trial attempts evaporate when localStorage clears
- The `/notifications` link in NotificationBell is a hard 404 (F1.4)
- 714 of 797 catalog books open to an empty reader (F8.1)
- Account deletion leaves user data dangling in 15 social-layer tables (F3.1)

What should NEVER be touched in the demo (security):
- The `achievements`, `leaderboard`, and `community_activity` tables are wide-open to anonymous writes (F2.1, F2.2, F2.3) — anyone with the published anon key can rewrite the leaderboard. Not exploitable in a 5-minute demo, but if disclosed to a YC technical reviewer, it's a red flag.

The ratio is striking and consistent across the audit: **the infrastructure layer (Supabase schema, RLS policies, server actions, auth flow, email pipeline, build pipeline) is genuinely production-grade. The UI integration is half-finished.** Where the gap appears most starkly:
- Server actions for classrooms / assignments / grades / peer-review / announcements / connections / recommendations are excellent. None of them are wired into the live classroom detail page.
- The Trial overlay, the marginalia toggles, the reader settings, and the bookmark drawer are all UI-complete but persist to localStorage instead of the DB tables that don't exist yet.
- Virgil-the-LLM is a stub from top to bottom: no API route, no system prompt, no rate limiter, no dossier, no chat history. The drawer that opens is an annotation-display drawer with a disabled chat footer.

### Top 5 things to fix before YC

1. **Tighten the three open-RLS policies (F2.1, F2.2, F2.3).** ~15 minutes of SQL. Drop `Allow * inserts/updates` on `achievements`, `leaderboard`, `community_activity`; replace with `auth.uid()`-scoped predicates. Survives any technical-reviewer security spot-check.

2. **Wire the classroom detail page to the live action layer (F5.1).** The actions exist, the schema exists, RLS exists. The page just calls demo data. Replace the `getClassroom/getAssignmentsForClass/getStudentsForAssignment` calls with `listAssignmentsForClassroom/listAnnouncements/listGradesForClassroom`. Highest visible-impact-per-hour fix in the audit.

3. **Fix the four cosmetic landing-flow leaks (F1.4, F1.8, F6.5, F7.2).** Remove `/notifications` link or ship the page; change LandingNav "Log in" → `/login` and "Sign Up" → `/signup`; change LandingFooter `/quiz` → `/quizzes`. Each is a one-line change. All are 404/redirect bounces visible to anonymous YC visitors.

4. **Decide on the homepage variant (F6.1).** The current `/` is the fifth marketing-homepage variant ("We are in a reading crisis" hero); the spec set has redesigned this four times. This is the screen every YC reviewer hits first — pick one and lock it. Not a bug, but a stakeholder decision that's been deferred.

5. **Either ship Virgil chat or remove the chat footer (F4.1).** Today the drawer renders a visibly-disabled chat input, which prompts the question "what does that do?" mid-demo. Either: (a) build the route + system prompt + Anthropic streaming + persistence (multi-day, blocked on F2.4 tables), or (b) hide the footer and call the drawer a "scholarly annotation drawer" — the curated annotations stand on their own. Option (b) is a five-minute fix and avoids the broken-promise framing.

Honorable mention (deferrable but compounds risk):

### 2.2 RLS policy posture

Every table has RLS enabled and at least one policy. No `relforcerowsecurity = true` (so service-role still bypasses, which is the intended pattern). However, three tables have **wide-open public-write policies** that survived from the earliest seed migrations:

| Table | Policy | `qual` | `with_check` | Effect |
|---|---|---|---|---|
| `achievements` | `Allow achievement inserts` (INSERT, public) | — | `true` | Any authenticated or anonymous user can insert any row |
| `achievements` | `Allow achievement updates` (UPDATE, public) | `true` | — | Any user can update any row |
| `leaderboard` | `Allow leaderboard inserts` (INSERT, public) | — | `true` | Same |
| `leaderboard` | `Allow leaderboard updates` (UPDATE, public) | `true` | — | Same |
| `community_activity` | `Allow activity inserts` (INSERT, public) | — | `true` | Any user can post arbitrary "activity" |

By contrast, `user_stats`, `db_notifications`, and the social-layer tables (connections, classrooms, etc.) are correctly scoped via `auth.uid()` predicates.

### 2.3 Schema observations worth noting

- **Two parallel "trial" models coexist.** Legacy: `quizzes` (27) → `questions` (135), strictly 4-option multiple choice (option_a/b/c/d/correct_option). New: `trials` (186) keyed on `section_id` (text) under the structured-content model (`works` → `sections` → `lines` + `trials` + `glosses` + `scene_annotations`). The `assignments` table has both `quiz_id (uuid)` and `trial_id (text)` columns referencing each.
- **`questions` table cannot represent 5 of 7 spec'd Trial question types.** Schema is MC-only; spec lists comprehension-mc, inference-mc, fill-blank, passage-id, thematic-matching, sequencing, reflection. The newer `trials` table uses `kind` + `options[]` so could fit more types depending on validator code, but `answer_index` is `int NOT NULL` which still implies single-choice-with-index.
- **Trial attempts are not persisted at all.** No `trial_attempts` table, no `trial_attempt_answers`. Anything the user does in `chapter-quiz-overlay.tsx` evaporates on refresh unless it's written to localStorage — which contradicts the spec's "auto-save via `saveAttemptProgress` server action."
- **`profiles` lacks the marginalia toggle columns** (`show_annotations`, `show_echoes`). Spec'd toggles in the reader settings have nowhere to persist.
- **`library_entries` has no progress/state columns.** Just (user_id, book_id, recommended_by, added_at). No `status`, `current_chapter`, `progress_percent`, `last_opened_at`. The reader has nothing to write back to mark progress.
- **`db_notifications` is named differently from spec.** Functionally equivalent (id, user_id, type, title, body, action_url, source_user_id, classroom_id, read, created_at). Code referencing `from('notifications')` would fail.

### 2.4 Findings

#### F2.1 — Wide-open INSERT/UPDATE on `achievements`
**Severity:** CRITICAL · **Scope:** 2 · **Files:** migration `20260330211651_create_achievements_table`
The policies allow any caller (`{public}`, `with_check: true`) to insert/update arbitrary rows. An attacker with the anon key can grant themselves any badge or overwrite anyone's achievements. Currently 0 rows so no exfil risk yet — but the surface is open.
**What I'd do about it:** drop both `Allow achievement inserts` and `Allow achievement updates`, then re-add scoped policies (e.g., `WITH CHECK (auth.uid() = user_id)`), or move all writes server-side and remove anon access entirely.

#### F2.2 — Wide-open INSERT/UPDATE on `leaderboard`
**Severity:** CRITICAL · **Scope:** 2 · **Files:** migration `20260330204152_create_user_stats_table` (or sibling)
Same pattern. 15 rows already, so any anon caller can rewrite them. Will publicly distort the leaderboard if discovered.
**What I'd do about it:** same remediation as F2.1; if leaderboard is computed server-side from `user_stats`, drop write policies entirely.

#### F2.3 — Wide-open INSERT on `community_activity`
**Severity:** CRITICAL · **Scope:** 2 · **Files:** migration `20260330210516_create_social_tables`
Anyone can insert arbitrary activity strings; will appear on community feeds.
**What I'd do about it:** restrict insert to `auth.uid() = actor_id` (or move writes server-side via service role).

#### F2.4 — Virgil persistence layer entirely missing
**Severity:** CRITICAL · **Scope:** 2/4 · **Files:** none — tables don't exist
Spec calls for `virgil_conversations`, `virgil_messages`, `virgil_usage_daily`, `reader_dossiers`. None exist. Without them: no chat history, no dossier (so the system prompt can't include "what does this user like"), no daily token cap (so `lib/virgil/rate-limit.ts` has nowhere to write its counter). Cross-references Scope 4 once we look at the code.
**What I'd do about it:** add a single migration creating all four tables with RLS scoped to `auth.uid() = user_id`. Defer the dossier-generation job; the schema can land first.

#### F2.5 — Trial attempts not persisted
**Severity:** HIGH · **Scope:** 2/4 · **Files:** none — tables don't exist
No `trial_attempts` / `trial_attempt_answers`. The chapter-quiz overlay therefore cannot resume in-progress attempts across reloads, classroom assignments cannot reference Trial scores, and no analytics on which questions trip students up. Reflection grading via Anthropic also has nowhere to store the LLM rubric output.
**What I'd do about it:** add a migration scoped to user_id with FK to `trials.id` (text) and a jsonb `answers` column, mirroring the existing `assignment_submissions` shape.

#### F2.6 — `profiles` missing `show_annotations` / `show_echoes`
**Severity:** HIGH · **Scope:** 2/4 · **Files:** migration `20260411000716_create_profiles_table`
The spec's reader settings let the user toggle marginalia visibility, persisted to `profiles`. Columns don't exist, so the toggles either silently don't persist or only update local component state.
**What I'd do about it:** ALTER TABLE adding `show_annotations boolean default true`, `show_echoes boolean default true`. Tiny migration, no data risk.

#### F2.7 — `library_entries` has no reading-state columns
**Severity:** HIGH · **Scope:** 2 · **Files:** spec'd schema vs migration `20260426163836_social_layer`
There's nothing to track which chapter a user is on, when they last opened a book, or whether they've finished it. The `/library` page can show "books on shelf" but cannot show "currently reading," "completed," or "X% through." Demo-relevant: the YC pitch likely references progress.

## Methodology Notes

### What was actually done

- **Branch:** `tome-recovery`. Working tree was dirty at audit start (~50 modified files, ~30 new files per `git status`). All findings reflect the on-disk state, not just the last clean commit.
- **Read-only:** no source files modified. Only writes were this report file. No migrations applied. No `git` mutations. No deployments.
- **Live database:** Scope 2 and Scope 8 hit the live Supabase project `vjaezrcuuzmbmnsfrtwt` (tome-app, us-east-1, Postgres 17.6) via the MCP server. Read-only queries: `list_tables`, `list_migrations`, plus 8 `execute_sql` SELECTs against `pg_policies`, `information_schema.columns`, the `books`/`chapters`/`annotations`/`works`/`sections`/`trials`/`lines`/etc. tables, and FK-integrity checks. No INSERT/UPDATE/DELETE.
- **Build pipeline:** ran `npx tsc --noEmit`, `npm run lint`, and `npx next build` in Scope 7. The lint command crashed with V8 OOM (F7.3) — flagged but did not retry under increased heap. The build was run with `NODE_OPTIONS="--max-old-space-size=8192"` and succeeded.
- **No dev server:** I did not run `npm run dev`. Broken-link findings (F1.4, F7.2) are static-analysis only and may miss runtime-only paths. I also did not interactively click through any UI surface; UI findings are based on file reads.
